## 2. The Three-Bucket System

Every contribution to this project falls into exactly one of three buckets. The bucket must be declared at the time of contribution. If no bucket is declared, the contribution defaults to Bucket 1 (Free).

### Bucket 1 -- Free (Donated)

The contributor donates work to the project. The project receives a permanent, irrevocable, worldwide, royalty-free license to use, modify, distribute, and sublicense the work as part of the project. The contributor retains credit for authorship (visible in git history) and retains the right to use their own contributed code in their own separate projects (dual license). Neither party can revoke this arrangement. The project keeps the work. The contributor keeps a copy.

**When to use Bucket 1:** You want to help the project, you believe in the mission, and you are giving freely. This is the default for open-source-style contributions.

### Bucket 2 -- Paid (Contract Work)

The contributor performs work under a written agreement that is signed BEFORE work begins. That agreement must specify: scope of work, price, deliverables, timeline, and payment terms. Intellectual property transfers to the project upon full payment. If payment is not made, the work never enters the repository and the contributor retains all rights. Partial payment disputes are resolved by the written agreement's terms.

**When to use Bucket 2:** You are being hired to build something specific. There is a price, a scope, and a deadline. Both parties sign off before any code is written.

**Required:** A separate written contract (even a simple email exchange with explicit terms counts) before work begins. Verbal agreements are not sufficient for Bucket 2 work.

### Bucket 3 -- Collaborative (Built Together)

Work that is genuinely created together -- the Owner and Contributor working in the same codebase, building on each other's commits, creating something neither could have built alone. The project receives a permanent license to use the work. Both parties receive credit. The work stays with the project if either party leaves. Neither party can sell the collaborative work independently or license it to a third party outside the project.

**When to use Bucket 3:** You and the Owner are pair-programming, co-designing, or building something intertwined. The git history shows meaningful commits from both sides.

## 3. What Contributors Cannot Do

These restrictions apply to all contributors regardless of bucket:

1. **No selling project assets.** You cannot sell, license, sublicense, or list for sale any project code, content, design, or infrastructure.

2. **No personal payment links on shared pages.** You cannot add PayPal links, Venmo links, donation buttons, tip jars, crypto addresses, affiliate links, or any other personal monetization mechanism to any shared project page without explicit written approval from the Owner.

3. **No unauthorized autonomous AI agents.** You cannot run AI bots, automated agents, or autonomous tools against the production repository without explicit written approval from the Owner. This includes but is not limited to: AI coding assistants configured to auto-commit, bots that generate pull requests, automated issue creation tools, and scheduled scripts that modify project files.

4. **No modifying governance files.** You cannot modify LICENSE, CODEOWNERS, .github/workflows, CONTRIBUTOR_LICENSE_AGREEMENT.md, or any governance/infrastructure configuration without the Owner's explicit approval.

5. **No ownership claims.** You cannot claim ownership of the project, any shared infrastructure, or any Bucket 3 (collaborative) work. Credit is not ownership.

6. **No external script injection.** You cannot add scripts, iframes, tracking pixels, or any resource that loads from a domain not controlled by the project without explicit written approval.

7. **No unauthorized repo syncing.** You cannot sync, mirror, or push content from a personal repository into the shared project repository without review and approval.

8. **No circumventing review.** You cannot push directly to production branches. All contributions go through pull request review.

## 5. AI and Bot Policy

Artificial intelligence tools are welcome in this project. They are also your responsibility.

1. **You own what your AI produces.** If you use an AI coding assistant, an LLM, a code generator, or any automated tool to create a contribution, that contribution is yours. It falls under whatever bucket you declared. "My AI did it" is not a defense for policy violations.

2. **Autonomous agents require written approval.** Any tool that can create commits, open pull requests, modify files, or interact with the repository without your real-time supervision requires explicit written approval from the Owner before it is given access. This approval can be revoked at any time.

3. **All bot-generated PRs require human review.** No automated tool may merge its own pull requests. A human (either the contributor or the Owner) must review and approve every merge.

4. **Volume limits.** If an AI tool generates an unreasonable volume of pull requests, issues, or commits that creates noise or disrupts the project, the Owner reserves the right to close them in bulk and revoke the tool's access.

5. **No AI-generated spam.** Contributions must be meaningful. Automated commits that pad statistics, generate empty files, or create busywork are not contributions -- they are noise, and they will be reverted.

## 6. Clean Exit Protocol

Anyone can leave at any time. Here is how it works:

1. **30-day notice preferred.** Not required, but appreciated. It gives both parties time to transition cleanly.

2. **Pre-existing work** (work you created before joining the project, provable via git timestamps or other documentation) remains entirely yours. The project has no claim to it.

3. **Bucket 1 (Free) work** stays with the project under the permanent license you granted. You also keep a copy and can use it in your own projects.

4. **Bucket 2 (Paid) work** follows the written contract. If the project paid for it, the project owns it. If the project did not pay for it, you keep it and it is removed from the repository.

5. **Bucket 3 (Collaborative) work** stays with the project. You keep credit. You keep a copy for reference. Neither party can sell it independently.

6. **Git history is permanent.** Your name, your commits, your contributions -- they stay in the record forever. We do not rewrite history.

7. **Access is revoked on departure.** Repository write access, deployment keys, API tokens, and any other project credentials are revoked when you leave. This is standard practice, not personal.

8. **No non-competes.** You can go build whatever you want after leaving. The only restriction is that you cannot take Bucket 2 (paid, transferred) or Bucket 3 (collaborative) work and use it in a competing project.
